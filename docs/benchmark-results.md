# Benchmark Results

This is real, reproducible output from `okpf benchmark`, not a claimed or
hand-written number. Regenerate it yourself:

```bash
PYTHONPATH=reference/python python3 -m okpf benchmark examples/software-onboarding
```

See [docs/benchmark-plan.md](benchmark-plan.md) for the methodology
(Q1 attribution, Q3 ingestion ambiguity, Q7 validator-caught errors) and
[docs/rag-export.md](rag-export.md) for how attribution/lineage are
computed per row. `okpf benchmark` measures the pack against naive
alternatives generated by the existing `okpf compare-layout` command — a
flat folder of Markdown files, and a single merged JSONL file with no
manifest — rather than a second, hand-maintained comparison
implementation, so this always reflects what `compare-layout` actually
produces.

## Reading these numbers honestly

- **Attribution completeness** and **source lineage completeness** are
  computed from `okpf export-rag`'s output: a row counts as attributed if
  it carries a non-empty `license` and `citation`, and as having lineage if
  its `provenance` field is non-empty (not `{}`). Neither naive alternative
  carries any of this — a flat Markdown folder has no metadata at all
  beyond file content, and a merged JSONL file only carries each record's
  own fields.
- **Ingestion decisions requiring guessing** counts how many of 5 fixed
  questions (package identity, license, domain, version, AI/RAG usage
  permission) can be answered by inspection alone, without asking the
  author. Note that JSONL-only isn't 5-for-5 guessing: `domain` is a
  *required* field on every OKPF record, so it survives even a maximally
  naive JSONL export. That's not a rounding error — it's the honest result
  of the metric actually inspecting the naive export's content rather than
  assuming it.
- **Validator-caught structural issues** intentionally does not report `0`
  for the naive alternatives. Reporting `0` would imply "zero errors
  found," which reads as *better* than an imperfect but real validator
  result — the honest statement is that no validator exists for either
  naive format at all.

## What this does not measure

Retrieval accuracy and hallucinated-source-citation rate (`docs/benchmark-plan.md`
Q4 and Q5) require a working RAG pipeline and an evaluation judge, which
this repository does not implement. `okpf benchmark` does not report
numbers for those questions — see `docs/benchmark-plan.md` for why they
remain marked "future work."

## Other examples

The numbers above are pack-specific, not a universal OKPF score — run the
same command against any example (or your own pack) to see its own
results:

```bash
PYTHONPATH=reference/python python3 -m okpf benchmark examples/local-history-lost-electric-sign
PYTHONPATH=reference/python python3 -m okpf benchmark examples/hello-world
```

A pack with no `records` at all (only `artifacts`) will show `0%` source
lineage completeness unless its `provenance` is populated — that's a
property of the pack's own authoring, not a benchmark quirk.
