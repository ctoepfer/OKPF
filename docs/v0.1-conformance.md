# OKPF v0.1 Conformance

This document defines what it means for producers and consumers to conform to OKPF v0.1.

Conformance is organized into four levels. A tool that satisfies a given level MUST satisfy all lower levels. Each level builds on the previous one.

Conformance to any level does not imply that pack contents are factually correct, safe, legally compliant, or operationally sound. OKPF validation verifies structure and declared integrity, not truth, quality, or fitness for purpose. Consumers MUST NOT treat a conformant pack as certified expertise.

OKPF does not define runtime execution semantics. A conformant tool reads and validates packs; it does not execute, interpret, or act on pack content unless the consuming application explicitly does so in its own context.

---

## Conformance Levels

### Level 0 — Manifest Reader

A Level 0 consumer CAN:

- Locate `manifest.json` at the root of a pack directory or at the root entry of a `.kpack` ZIP archive.
- Parse `manifest.json` as UTF-8 JSON.
- Read and use the following required fields when present: `okpf_version`, `name`, `version`, `domain`, `license`.
- Tolerate unknown or optional fields — unknown fields MUST NOT cause the consumer to reject the pack or raise an error.
- Identify declared entries in `artifacts`, `records`, and `content` arrays.
- Reject obviously unsafe artifact or entry paths before reading archive entries. Unsafe paths include:
  - Absolute paths (beginning with `/` or a Windows drive letter such as `C:\`)
  - Parent directory traversal (containing `..`)
  - Backslash sequences
  - NUL bytes

A Level 0 consumer SHOULD NOT require network access to read a pack.

A Level 0 consumer SHOULD NOT auto-execute pack content.

---

### Level 1 — Core Validator

A Level 1 consumer CAN perform all Level 0 behavior, and additionally:

- Validate `manifest.json` against the local OKPF v0.1 manifest schema (`schemas/v0.1.0/manifest.schema.json`) without fetching remote schemas.
- Validate each declared record file against the local record schema (`schemas/record.schema.json`) when records are declared and the record format is `json` or `jsonl`.
- Confirm that each artifact or record path in the manifest is a safe relative path (Level 0 requirement), then verify that the file actually exists in the pack.
- Confirm that `okpf_version` is `"0.1"` or `"0.1.0"`.
- Confirm that at least one of `artifacts`, `records`, or `content` is declared and non-empty.
- Distinguish validation errors (pack is structurally invalid) from advisory warnings (pack is valid but missing recommended metadata).
- Emit warnings for missing recommended fields: `description`, `creators`, `usage_policy`, `provenance`.
- Never require network access for core validation.

A Level 1 consumer MUST NOT reject an otherwise valid pack solely because it contains fields the consumer does not recognize.

A Level 1 consumer MUST NOT fetch remote schemas, registries, or external resources during validation.

---

### Level 2 — Integrity-Aware Consumer

A Level 2 consumer CAN perform all Level 1 behavior, and additionally:

- Verify declared SHA-256 hashes for artifacts when `integrity.artifacts` is present in the manifest.
- Report missing hashes as warnings and mismatched hashes as errors.
- Preserve package identity (`package_id` or `id`), version, license, provenance metadata, and attribution when ingesting or re-exporting pack content.
- Distinguish integrity verification failures (hash mismatch) from structural validation failures (schema violation) in output and return codes.

A Level 2 consumer MUST verify hashes by reading file content locally — it MUST NOT fetch file content from remote sources unless the user explicitly requests networked behavior.

---

### Level 3 — Profile-Aware Consumer

A Level 3 consumer CAN perform all Level 2 behavior, and additionally:

- Recognize profile identifiers declared in `manifest.json#/profiles`.
- Apply local profile schemas or rule sets when available for a recognized profile.
- Emit warnings when a declared profile is unrecognized and no local schema is available, unless strict profile validation is not requested.
- Treat profile validation failures as warnings by default, escalating to errors only when strict profile mode is explicitly enabled.
- Avoid treating profile-specific fields or conventions as OKPF Core requirements.
- Validate fermentation-profile records against `profiles/fermentation/v0.1.0/` schemas when the `okpf-fermentation` or `fermentation` profile is declared.

A Level 3 consumer MUST NOT reject a pack as invalid at the core level solely because it declares an unknown profile.

A Level 3 consumer SHOULD NOT require remote access to load profile schemas.

---

## Producer Conformance

A conformant producer MUST:

- Include `manifest.json` at the pack root.
- Include all required manifest fields: `okpf_version`, `name`, `version`, `domain`, `license`, and one of `package_id` or `id`.
- Set `okpf_version` to `"0.1"` or `"0.1.0"`.
- Include at least one of `artifacts`, `records`, or `content` as a non-empty array.
- Use only safe relative paths for artifact, record, and source entries.
- Declare the correct MIME type for each artifact when known.
- Use UTF-8 encoding for all text files.

A conformant producer SHOULD:

- Include `description`, `creators`, `usage_policy`, and `provenance`.
- Declare SHA-256 hashes for artifacts when integrity verification is desired.
- Use `package_id` rather than the legacy `id` field for new packs.
- Declare applicable profiles in `manifest.profiles` rather than embedding profile-specific fields in the core manifest.
- Preserve unknown optional fields when reading and rewriting a manifest.

A conformant producer MUST NOT:

- Use absolute paths, parent traversal, backslash sequences, or NUL bytes in artifact or record paths.
- Include paths that could escape the pack root during extraction.
- Auto-execute content during pack construction.
- Fetch remote resources during pack construction.
- Embed credentials, private API keys, private absolute filesystem paths, or personally identifiable information in schemas, examples, or manifests.

## Error vs Warning Guidance

| Condition | Severity |
|-----------|----------|
| Missing required field (`okpf_version`, `name`, `version`, `domain`, `license`, `package_id` or `id`) | Error |
| No payload declared (`artifacts`, `records`, `content` all absent) | Error |
| Invalid `okpf_version` value | Error |
| Unsafe artifact or record path | Error |
| Declared artifact or record file not found | Error |
| Invalid JSON in manifest or record file | Error |
| Schema validation failure | Error |
| SHA-256 mismatch (Level 2+) | Error |
| Unknown profile identifier (Level 3, default mode) | Warning |
| Profile-specific convention not followed (Level 3, default mode) | Warning |
| Missing recommended field (`description`, `creators`, `usage_policy`, `provenance`) | Warning |
| Hash not declared for an artifact (no `integrity`) | Advisory / no output |

## Required vs Recommended Behavior

| Behavior | Required (MUST) | Recommended (SHOULD) |
|----------|----------------|---------------------|
| Parse UTF-8 `manifest.json` | ✓ | |
| Tolerate unknown fields | ✓ | |
| Reject unsafe paths | ✓ | |
| Validate against local schema | ✓ (Level 1+) | |
| Never fetch remote schemas | ✓ | |
| Never auto-execute pack content | ✓ | |
| Preserve provenance/license on ingestion | ✓ (Level 2+) | |
| Verify SHA-256 hashes when declared | ✓ (Level 2+) | |
| Warn on unrecognized profiles | | ✓ (Level 3) |
| Report missing recommended metadata | | ✓ (Level 1+) |

## Explicit Scope Limitations

The following are out of scope for OKPF conformance at any level:

- **Factual correctness.** Conformance does not imply that pack content is accurate, current, or expert-reviewed.
- **Safety.** Conformance does not imply that following pack content is safe for any purpose.
- **Regulatory compliance.** Conformance does not constitute legal, medical, engineering, or safety certification.
- **Usefulness.** Conformance does not guarantee that a pack is useful for any particular application.
- **Runtime semantics.** OKPF does not define how pack content is executed, interpreted, retrieved, or applied.
- **Blockchain or signature verification.** Anchors and signatures are optional; their absence does not affect core conformance.
- **Embeddings.** Pre-computed vector embeddings are optional; their absence does not affect core conformance.

## Reference Implementation

The standalone validator at [reference/python/okpf_validate.py](../reference/python/okpf_validate.py) implements Level 2 behavior with optional Level 3 profile validation for the fermentation profile.

```bash
# Core validation (Level 2)
python3 reference/python/okpf_validate.py examples/hello-world

# Profile-aware validation (Level 3)
python3 reference/python/okpf_validate.py examples/fermentation-mixed-bundle --profile fermentation

# Strict profile mode (Level 3, warnings as errors)
python3 reference/python/okpf_validate.py examples/fermentation-mixed-bundle --strict-profile
```

The reference CLI at [reference/python/okpf/](../reference/python/okpf/) exposes `validate`, `inspect`, `pack`, and `unpack` commands:

```bash
PYTHONPATH=reference/python python3 -m okpf validate examples/hello-world
PYTHONPATH=reference/python python3 -m okpf inspect examples/hello-world
PYTHONPATH=reference/python python3 -m okpf pack examples/hello-world out/hello-world.kpack
PYTHONPATH=reference/python python3 -m okpf unpack out/hello-world.kpack out/hello-world-unpacked
```
