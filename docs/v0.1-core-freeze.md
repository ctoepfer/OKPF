# OKPF Core v0.1 — Freeze Declaration

**Status: Core frozen.** This document declares OKPF Core v0.1 frozen for
required-field changes, and states the discipline the project commits to
for everything that comes after. It doesn't introduce new schema, new
required fields, or new validation behavior — it's a governance act, not
a spec change.

## Why now

[docs/phase-1-roadmap.md](phase-1-roadmap.md) and the project's internal
adoption plan set a concrete bar before declaring Core stable: eight
"minimum viable boringness" criteria. All eight were re-verified against
a genuinely fresh `git clone` (fresh venv, editable install, no prior
state) as part of this freeze:

| Criterion | Verified by |
|---|---|
| A new user can create a valid pack in under 10 minutes | `okpf init --template ...` + `okpf validate` — seconds, not minutes |
| A developer can validate a pack without reading the schema | `okpf validate <pack>` |
| The README quickstart works from a fresh install | Every documented command in `README.md`, run verbatim from a fresh clone |
| Every example validates and pack/unpack round-trips in CI | `tools/ci_check_examples.py`, wired into `.github/workflows/ci.yml` |
| AI/RAG export produces stable, predictable fields every time | `okpf export-rag`, deterministic `chunk_id`/`sha256` (docs/rag-export.md) |
| Legacy `id`/`content` fields warn clearly but do not break old packs | `okpf validate`/`okpf explain` on `tests/fixtures/conformance/valid/legacy-*` |
| No required field needs philosophical explanation | Core required fields: `okpf_version`, `name`, `version`, `domain`, `license`, `package_id`/`id`, one of `artifacts`/`records`/`content` — all self-explanatory |
| Validation errors include actionable fix suggestions | `okpf explain <pack>` |

Fixing this freeze also fixed a real bug the audit turned up: the root
`README.md` contained a stale hardcoded local path
(`cd /home/toepf/Projects/OKPF`) in its own quickstart, which would have
failed the "works from a fresh install" criterion for anyone who actually
tried it verbatim. That's fixed as part of this freeze, not swept under
it — a freeze declaration that didn't survive contact with a fresh clone
wouldn't be worth making.

## What is frozen

As of this declaration:

- **`schemas/v0.1.0/manifest.schema.json`** and the required fields in
  [SPEC.md](../SPEC.md) will not be renamed or removed without a major
  version bump and a migration guide, per the stability commitment
  already stated in [docs/phase-1-roadmap.md](phase-1-roadmap.md).
- **The `id`/`content` compatibility aliases are permanent for v0.1.x.**
  They are not deprecated-and-pending-removal; packs using them are and
  will remain fully valid. (The migration story's `v1.0 decision: TBD`
  on whether aliases become a permanent guarantee is unaffected — that's
  a v1.0 question, not a v0.1 one.)
- **No new required manifest field will be added within v0.1.x.** A new
  required field is a breaking change in spirit, per
  [docs/phase-1-roadmap.md](phase-1-roadmap.md)'s stability commitment,
  regardless of whether it's technically additive.
- **Optional fields remain free to evolve** additively, per SPEC.md's
  existing "Compatibility and Versioning" section — this freeze does not
  slow that down. Adding an optional field, or clarifying the semantics
  of a low-adoption optional field, is not a Core-freeze violation.

## What stays an optional extension, permanently

Restating and consolidating what several docs already say individually,
because a freeze is exactly the moment scope creep becomes tempting:

- **Embeddings and semantic retrieval** — optional, model/provider-neutral, never required for a pack to be valid. See [docs/future-knowledge-sources.md](future-knowledge-sources.md) §4/§8 for where synthetic/AI-adjacent conventions are headed, none of it in Core.
- **Signatures and cryptographic attestation** — sketched in SPEC.md as "Future: Signatures and Cryptographic Seals," still future, still optional.
- **Registries and discovery indexes** — see [docs/registry-index.md](registry-index.md): "OKPF itself is not a marketplace or registry."
- **Blockchain anchoring** — see [docs/blockchain.md](blockchain.md): fully optional, "a knowledge pack without any blockchain anchors is fully conformant and fully functional."
- **Marketplace, payments, DRM, royalty/licensing enforcement mechanics** — not in Core, not planned for Core.
- **Any specific AI provider, vector database, or runtime dependency** — Core validation stays offline-capable and infrastructure-neutral (see root `CLAUDE.md` Security Rules).

## Profiles stay optional, not a gate

Per [docs/profile-authoring.md](profile-authoring.md), profiles MUST NOT
introduce new Core-required fields, mandate infrastructure, or redefine
Core semantics. Freezing Core does not change this — if anything, freezing
Core makes profiles the *only* correct place for a domain to formalize its
own conventions going forward (see the `empirical-discovery` and
`synthetic-knowledge` profile candidates named in
[docs/future-knowledge-sources.md](future-knowledge-sources.md) §8, and
the existing [fermentation](../profiles/fermentation/v0.1.0/),
[physical-skill-evidence](../profiles/physical-skill-evidence/v0.1.0/), and
[human-correction-loop](../profiles/human-correction-loop/v0.1.0/)
profiles as the working model). A pack that ignores every profile must
remain exactly as valid as one that uses one.

## How a Core change would happen anyway

Freezing isn't a promise that Core is perfect — it's a promise that
changing it now costs more than it used to, on purpose. A required-field
change to v0.1.x Core would need, at minimum:

1. **Adoption evidence**, not a hypothetical — per
   [docs/adoption-strategy.md](adoption-strategy.md)'s evidence log:
   real external usage surfacing a concrete gap that an optional field or
   a profile genuinely cannot solve.
2. **A public proposal**, per [GOVERNANCE.md](../GOVERNANCE.md)'s
   openness and stability principles — documented rationale, open to
   community input, not a unilateral schema edit.
3. **A compatibility story** — old valid packs must not become invalid;
   see the migration-story pattern already used for `id`/`content` in
   [docs/phase-1-roadmap.md](phase-1-roadmap.md) as the template.
4. **A major or minor version boundary**, per SPEC.md's semantic
   versioning commitment — never a silent change to `0.1.0`'s meaning.

## What this freeze does not mean

- It does not mean OKPF Core is bug-free — conformance and validator bugs
  still get fixed as bugs, same as always.
- It does not mean documentation, examples, tooling (`okpf` CLI), or
  reference SDKs are frozen — Phases 2–5 of the adoption plan (RAG export,
  benchmarking, packaging, adoption outreach) all continue.
- It does not mean profiles or extensions stop evolving — they're
  explicitly where evolution is supposed to happen now.
- It does not mean the project stops listening to adoption feedback — see
  [docs/adoption-outreach-kit.md](adoption-outreach-kit.md). Feedback is
  exactly what could justify reopening this freeze later, deliberately,
  through the process above — not around it.

---

> A small, stable, hand-inspectable Core is the whole point. Everything
> interesting is allowed to keep changing — as long as it changes outside
> Core, where breaking it doesn't break every pack that came before.
